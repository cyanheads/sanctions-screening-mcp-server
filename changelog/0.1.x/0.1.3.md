# 0.1.3 — 2026-06-19

## Fixed

- **`sanctions_screen_name` minScore floor** — removed the `|| phoneticHit` bypass in `ScreeningService.runFuzzy`; every fuzzy candidate is now admitted only when `score >= minScore`, so no sub-floor hit is returned. Phonetic-key candidates still seed the pool, but face the same floor as token/Jaro-Winkler candidates. `minScore` `.describe()` updated to state the floor applies uniformly across match strategies. ([#1](https://github.com/cyanheads/sanctions-screening-mcp-server/issues/1))
- **Mirror lifecycle scripts emitted no output** — `bootstrap()` in `scripts/_mirror-context.ts` is now `async` and calls `await logger.initialize(config.logLevel, 'stdio')` before returning. The framework logger drops all `log()` calls until `initialize()` runs, which the lifecycle scripts (`mirror-init`, `mirror-refresh`, `mirror-verify`, `mirror-seed`) bypassed by not going through `createApp()`. ([#3](https://github.com/cyanheads/sanctions-screening-mcp-server/issues/3))
- **`screen_nodes` → `screenNodes`** — corrected every doc surface that named the `sanctions_trace_ownership` parameter in snake_case (tool description, two output `.describe()` strings, the `sanctions_vet_counterparty` prompt, README, `docs/design.md`, and the agent protocol files) to match the actual camelCase Zod parameter. A model following the old docs passed an unknown key that Zod dropped, silently skipping beneficial-ownership screening. ([#2](https://github.com/cyanheads/sanctions-screening-mcp-server/issues/2))
